# Kanon Core Philosophy & Roadmap

このドキュメントは、Kanon（arahabaki）が目指す「プロジェクトを自律的に育てるAI」としての基本設計思想、教育の階層構造、および自己拡張ループに向けた開発スプリントを定義した最上位のロードマップです。

---

## 🧭 1. Kanonの基本哲学 (Core Philosophy)

Kanonは、単なる「人間の指示に従ってコードを書く便利ツール」ではありません。  
**「人間と協調しながら、自ら課題を発見し、プロジェクトを主体的に育てる開発パートナー」** を目指します。

### 🧠 GitHub ＝ AIの外部記憶装置（脳）
GitHubを単なるソースコードの置き場所ではなく、**Kanonの長期記憶・認知機能の一部**として位置づけます。

* **Issue** ➔ **未来の仕事（タスク・課題）**
* **Pull Request (PR)** ➔ **実施内容（行動履歴）**
* **Review** ➔ **人間からの教育・フィードバック**
* **Discussion** ➔ **思考ログ・対話ログ**
* **Wiki** ➔ **体系化された知識**
* **Projects** ➔ **現在の脳（タスクの優先順位と状態）**

## 🏫 2. 教育の3層構造 (3-Layer Education)

Kanonの知性と行動指針を以下の3つのレイヤーで整理し、継続的にアップデートします。

```
+-------------------------------------------------------+
|  【第一層】 憲法 (Constitution)                         |
|   - AGENTS.md, README.md, ロードマップ                 |
|   - 普遍的な原則、安全境界（ASEP）、Git運用ルール        |
+-------------------------------------------------------+
                           ▲ 振り返りからアップデート
+-------------------------------------------------------+
|  【第二層】 記憶・経験 (Memory/Experience)             |
|   - 意味記憶、エピソード記憶 (pytestの失敗と解決策など)    |
|   - 人間からの指摘、失敗した設計の反省ログ               |
+-------------------------------------------------------+
                           ▲ 抽出・知識化
+-------------------------------------------------------+
|  【第三層】 仕事 (Tasks/Inbox)                         |
|   - 未解決のIssue、READMEの更新候補、テスト不足の検知  |
|   - Discordログやメモから抽出された「今やるべきタスク」|
+-------------------------------------------------------+
```

---

## 📥 3. コア機能：インボックス (Inbox) の概念

対話を「仕事」へと変換するための情報流入ゲートウェイとして **`Inbox`** を設置します。

1. **インプット**: `/import` コマンド等により、Discordの過去ログ、打ち合わせのメモ、PDF、音声文字起こし等のあらゆる雑多な情報をインボックスに投入。
2. **アイドリング解析**: Kanonが待機時間（暇なとき）にインボックスを自律的にパース。
3. **仕事化ループ**:
   * ログから「知識」を抽出 ➔ `Memory` へ蓄積。
   * 課題や不足（テスト不足、ドキュメントの古さ）を検知 ➔ `Issue`（未来の仕事）を自動起票。
   * 人間に「これを進めてよいか？」と相談（DiscordやCLI上で提案）。

## 🗺️ 4. 開発ロードマップ (Sprints)

Kanonの「自己拡張ループ」を成立させるため、以下の6つのスプリントに分けて開発を推進します。

```
+-----------------------------------------------------------------------------------+
|  Sprint 1 (実行基盤)  ➔  GitHub連携 (Branch/Commit/PR/Issue自動起票のAPI実装)     |
+-----------------------------------------------------------------------------------+
                                         ▼
+-----------------------------------------------------------------------------------+
|  Sprint 2 (仕事化)    ➔  Task Manager (「今やる仕事」の優先順位管理・状態遷移)    |
+-----------------------------------------------------------------------------------+
                                         ▼
+-----------------------------------------------------------------------------------+
|  Sprint 3 (監視・観測) ➔  Observer (Discord/ログ/pytest結果のバックグラウンド監視) |
+-----------------------------------------------------------------------------------+
                                         ▼
+-----------------------------------------------------------------------------------+
|  Sprint 4 (プランニング) ➔  Planner (Inboxから「Issue候補」を自動起票する仕組み)   |
+-----------------------------------------------------------------------------------+
                                         ▼
+-----------------------------------------------------------------------------------+
|  Sprint 5 (実作業)    ➔  Executor (子エージェントや他のモデルにタスクを分散・実行)|
+-----------------------------------------------------------------------------------+
                                         ▼
+-----------------------------------------------------------------------------------+
|  Sprint 6 (学習・自己改変) ➔  Reflection (タスク終了時の振り返りとMemory蓄積)      |
+-----------------------------------------------------------------------------------+
```
