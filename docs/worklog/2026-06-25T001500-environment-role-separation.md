# ワークログ: 開発機（Mac）と検証機（GPD WIN 3）の物理的な役割分離とワークフロー設計

* **作成日時**: 2026-06-25T00:15:00+09:00
* **テーマ**: Kanon における開発環境（Mac/M4 Mac）と実機検証環境（GPD WIN 3/Ubuntu）の明確な用途分離、および GitHub を媒介とした非同期コラボレーション・フローのキャノニカライズ。

---

## 1. 🎯 目的と背景

Kanon の開発において、エージェントが検証用実機（GPD WIN 3 / Ubuntu）に直接 SSH 等で接続して修正と検証を繰り返すアプローチは、環境差異による破壊、ネットワーク依存、および接続セキュリティの観点から非効率かつハイリスクであることが判明しました。
これを受け、「用途分離」「役割固定」の原則に基づき、開発環境（Mac）と検証環境（GPD WIN 3）の物理的役割を完全に分離し、GitHub を仲介した健全な開発・検証ループを正式に固定化します。

---

## 2. 📐 環境マトリクスと役割分担

| 環境 (マシン名) | OS | 主な役割・責務 | エージェントの行動境界 |
| :--- | :--- | :--- | :--- |
| **開発環境 (M4 Mac)** | macOS | コードの修正・リファクタリング、設計書の整備、L1 単体テスト（pytest）の実行、および Git コミット・GitHub プッシュ。 | **AGY 稼働エリア**。コード編集とローカル検証はすべてここで行う。 |
| **検証環境 (GPD WIN 3)** | Ubuntu 26.04 | Docker ビルド、コンテナの起動検証、`/healthz` 疎通、OS固有のトラブルシューティング、ロールバックなどの実機検証（L2/L3）。 | **エージェントからの直接 SSH 接続は禁止**。人間（ユーザー）が手動で pull・検証を行う。 |
| **中継地点 (GitHub)** | - | 開発機と検証機間のソース同期。 | 双方の修正をマージ・統合する唯一の窓口。 |

---

## 3. 🔄 非同期開発・検証ワークフロー

```text
[AGY (M4 Mac)] ➔ コード修正・単体テスト (L1) ➔ Git Push
      ↓ (GitHub 同期)
[Human (GPD WIN 3)] ➔ Git Pull ➔ Docker 起動・実機検証 (L2/L3) ➔ 問題発見
      ↓ (フィードバック / Issue化)
[AGY (M4 Mac)] ➔ 再び修正・テスト ➔ Git Push (ループ継続)
```

この分離により、エージェントは GPD 固有の SSH 接続エラーなどに煩わされることなく、Mac 上で 100% のパフォーマンスでコードの磨き上げとテスト自動化に集中できます。また、実機固有の問題（`docker-proxy` や `apt-key` など）は、人間が検証段階で確実に検知して切り分けるため、システム全体の安全稼働（Survival First）が強固に維持されます。
