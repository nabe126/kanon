## 🧭 1. ゼロベース設計の意思決定：「AIエージェントを運営するOS (Kanon OS)」への昇華

本日はコードの実装変更は行わず、Kanonの本来の価値と将来性を担保するため、アーキテクチャの根本的なリファクタリング構想を「ネリネリ」しました。

これまでの「AIエージェント自身（ファイル操作やGit・PR起票など）を作る」という方向性をゼロベースで見直し、Kanonを **「AIエージェント（Worker）を運営するOS（またはPM/スクラムマスター）」** として完全に再定義しました。

---

## 🏛️ 2. 定義された7層アーキテクチャ

Kanonおよび周辺世界を以下の7つのレイヤー（またはコンポーネント）に抽象化し、それぞれの責務の境界線を明確にしました。

```
+-------------------------------------------------------+
|  【Human】                                            |
|    - 開発の指示者・最終承認者                          |
+-------------------------------------------------------+
                           │
                           ▼
+-------------------------------------------------------+
|  【Kanon Core (OS/PM)】                               |
|    - ① Inbox: 対話・ログ・Issue等のあらゆる情報流入   |
|    - ② Memory: 短期・長期・知識・反省ログの蓄積       |
|    - ③ Planner: 優先度決定、タスク分解、Workerアサイン  |
|    - ④ Policy: ASEP等の安全承認境界、Git運用ポリシー |
|    - ⑤ Scheduler: 朝会、定期レビュー催促、Memory整理   |
|    - ⑥ Review: 成果物の検収（合格判定・リテイク）      |
+-------------------------------------------------------+
                           │
                           ▼ タスク発注 (TaskDescriptor)
+-------------------------------------------------------+
|  【Executor (配送・回収)】                             |
|    - OSからの発注をWorkerへ配送し、結果を回収する     |
+-------------------------------------------------------+
                           │
                           ▼ アサイン (コマンドキック等)
+-------------------------------------------------------+
|  【Worker (実行エージェント / 手足)】                 |
|    - agy, Claude Code, Gemini CLI, Codex CLI, etc.    |
|    - ※ 最初のWorker実装として `agy` を利用            |
+-------------------------------------------------------+
                           │
                           ▼
+-------------------------------------------------------+
|  【World (外界)】                                     |
|    - GitHub, Discord, Docker, SSH, Filesystem, etc.   |
+-------------------------------------------------------+
```

## 🛠️ 3. 次回（Sprint 1）に向けた開発の方向性

Kanonの価値は「一番賢いこと」ではなく、**「プロジェクトの時間軸（数週間〜数ヶ月の流れ）を管理し、維持し続けること」** です。
そのため、次回の開発は「機能追加」ではなく、「境界線の定義とドキュメントリファクタリング」から開始します。

### ディレクトリ構成の最終形イメージ（コンポーネント化）
```
src/
└── kanon/
    ├── inbox/
    ├── planner/
    ├── memory/
    ├── policy/
    ├── scheduler/
    ├── executor/
    ├── worker/
    └── review/
```

この思想の言語化と設計図の作成こそが、Kanon v2の本当の土台となります。
