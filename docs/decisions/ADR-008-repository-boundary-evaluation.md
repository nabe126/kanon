# [ADR-008] 将来的なリポジトリ境界（Monorepo vs. Multi-repo）の評価

## ステータス
**提案中 (Proposed)** - 2026-06-20T21:43:24+09:00

---

## 1. 背景 (Context)

[ADR-007](file:///Users/nabe/src/github.com/nabe126/kanon/docs/decisions/ADR-007-kanon-core-twin-split-decision.md) において、現段階（Phase 0/1）では物理的なリポジトリ分離は行わず、単一リポジトリ内で論理的境界を維持する方針（Split Later）を決定しました。

本ドキュメントでは、将来的な Kanon の成長（長期記憶の蓄積、デジタルツインの稼働、外部システムとの協調）に伴って発生するリポジトリ境界の再編について、以下の3つの候補を多角的に評価し、移行条件と推奨時期を定義します。

* **候補 A**: monorepo 維持 (単一リポジトリですべて管理)
* **候補 B**: core と memory の分離 (2リポジトリ構成)
* **候補 C**: core / memory / workhub の分離 (3リポジトリ構成)

---

## 2. 候補の評価

### 候補 A: monorepo 維持 (単一リポジトリ)
プログラムコード、Markdown長期記憶、システム構成、ドキュメントをすべて1つの Git リポジトリで管理する構成。

* **メリット (Pros)**:
  * **極めて低いオーバーヘッド**: コミット、CI/CD、Docker ビルド、開発環境の立ち上げが1箇所で完結。
  * **整合性の保証**: コード変更とメモリ構造定義、またはルール変更が 1 コミットで同期されるため、バージョン不整合が起きない。
  * **ガバナンスの一元化**: 共通の `.agents/AGENTS.md` (憲法) のもとでエージェントに厳格なルールを適用しやすい。
* **デメリット (Cons)**:
  * **セキュリティ・権限分離の限界**: `memory/` に蓄積される個人のプライベート情報（health, family 等）と、コード開発（OSS化や外部との共同開発）のアクセス権限をリポジトリレベルで分離できない。
  * **容量の肥大化**: 記憶データ（特に画像などのバイナリを含むカメラログ等）が増大した際、コード開発時にリポジトリを軽量クローンすることが困難になる。
* **移行条件 (Aを抜ける条件)**:
  * 外部の共同開発者（他のAI、または人間）をコード開発に招待する要件が発生したとき。
  * `memory/` 内の個人情報をパブリックな Git リポジトリ（GitHubなど）から完全に切り離すセキュリティ・プライバシー要件が発生したとき。
* **推奨時期**:
  * **Phase 0 (Foundation) 〜 Phase 2 (Long-term Memory 初期段階)** まで。

---

### 候補 B: core と memory の分離 (2リポジトリ)
実行脳やツールエンジンを持つ `kanon-core` と、Markdownの長期記憶群を持つ `kanon-memory` の2つに分割する構成。

* **メリット (Pros)**:
  * **セキュリティとプライバシーの確保**: `kanon-core` はパブリック/共有リポジトリでオープン開発し、`kanon-memory` はローカルサーバーや暗号化されたプライベート Git のみで厳重に保護する運用が可能。
  * **履歴の分離**: メモリのインクリメンタルな頻繁な更新履歴と、システムとしてのリリース・開発履歴が混ざらず、履歴が綺麗に保たれる。
  * **データ保護**: コアコードの書き換え・適用ミス（自動ロールバックのバグ等）が、物理的に記憶（Memory）を破壊・削除するリスクを最小化。
* **デメリット (Cons)**:
  * **外部接続設定のオーバーヘッド**: エージェントコンテナから `kanon-memory` リポジトリへ SSH やアクセストークンを介して Pull/Push を実行する接続ハンドシェイクが追加され、複雑性が増す。
  - **構造バージョン管理の難化**: コア側が要求するメモリ形式と、実際のメモリの形式にズレが生じた場合の例外処理が必要。
* **移行条件 (Bへ移行する条件)**:
  * エージェントのコードベース（Kanon Core）を GitHub 等で OSS / パブリック公開する時。
  * `memory/` ディレクトリのファイルサイズが肥大化し、開発環境のクローン速度を低下させ始めた時。
* **推奨時期**:
  * **Phase 2 (Long-term Memory) の終盤から Phase 3 (Research Agent) の開始時**。

---

### 候補 C: core / memory / workhub の分離 (3リポジトリ)
実行エンジン `kanon-core`、記憶データ `kanon-memory` に加え、人間とエージェントが協調するためのポータル/Web UI/ダッシュボードである `kanon-workhub` を独立させる構成。

* **メリット (Pros)**:
  * **関心の完全分離**: UI（Next.js/Vite等）とエージェント実行ランタイム（Python）を完全に独立させることで、エージェントプロセスが何らかのエラーでクラッシュしている最中でも、ダッシュボードや人間用のポータルは生存し続け、エラー状況を人間がブラウザから確認・操作できる。
  * **マルチユーザー・コラボレーション対応**: 1つの Core エージェントだけでなく、複数の人間や複数のエージェントが同一の Workhub を通じてタスクやプロジェクトを一覧し、共同作業できる。
* **デメリット (Cons)**:
  * **アーキテクチャの急激な複雑化**: Core, Memory, Workhub 間での通信API（REST / GraphQL / WebSocket）のプロトコル策定、認証、セッション管理が必要となり、個人運用の範囲ではオーバーヘッドが大きすぎる。
* **移行条件 (Cへ移行する条件)**:
  * エージェントのダッシュボードを単なる Flask の静的ページから、高度な対話型タスクボードやコラボレーションポータルへと進化させる時。
  * エージェントが単一の「相棒」を超え、組織やグループで業務を処理するための共通フロントエンド（ハブ）が必要になった時。
* **推奨時期**:
  * **Phase 4 (Digital Twin) の確立後、または Phase 5 (Semi-Autonomous) 以降**。

## 3. 推奨ロードマップと意思決定マトリクス

| 開発フェーズ | 推奨リポジトリ境界 | 理由 |
| :--- | :--- | :--- |
| **Phase 0〜1 (生存基盤)** | **候補 A (Monorepo)** | 構造定義とインフラの動作検証を最速でループさせるため。 |
| **Phase 2〜3 (記憶・調査)** | **候補 B (Core/Memory 分離)** | 記憶データのプライバシー確保と、GitHub等へのCoreコードの共有・安全なOSS化を可能にするため。 |
| **Phase 4〜5 (ツイン・半自律)** | **候補 C (Core/Memory/Workhub)** | ユーザーとの業務協調UI（Next.js等のダッシュボード）を独立させ、エージェントの状態に関わらずポータルを稼働させるため。 |
