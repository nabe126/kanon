# [ADR-012] Executor 抽象化と PM（プロジェクトマネージャー）モデルの採用

## ステータス
**提案中 (Proposed)** (2026-07-12)

## 背景 (Context)
これまで Kanon (arahabaki) は、自律型開発エージェントとして自身の手足となる開発ツール（ファイル操作、Grep検索、Gitコミット、GitHub APIによるPR自動起票など）の機能をすべて自前（`agent_tools.py` や GitHub REST API）で実装・管理しようとしていました。

しかし、このアプローチには以下の重大な課題があります。
1. **車輪の再発明**: ファイル操作やGit操作、テスト実行、PR作成などのコード実行タスクは、すでに Google Antigravity の公式 CLI である `agy` などの既存ツールが高い完成度で実現しています。これらを Kanon 側でゼロから再実装・保守することは非効率であり、バグのリスクを増大させます。
2. **Kanon のコア価値の希薄化**: 汎用のコード編集機能に工数を割かれる結果、Kanon 本来の目的である「自律的な課題の発見」「長期記憶の整理」「プロジェクトの時間軸の維持（時間連続性）」といった「脳・マネジメント層」の機能開発が遅れてしまいます。
3. **拡張性と寿命の限界**: 編集機能と本体が密結合していると、将来的に `Claude Code` などの新しい優秀なコード編集ツールが登場した際、Kanon 本体を大幅に書き換える必要が生じます。

## 決定 (Decision)
Kanon を「コード編集を行うAI」ではなく、**「Worker（エージェント）を管理し、プロジェクトの時間軸を維持する PM (プロジェクトマネージャー) / OS」** として再定義し、実行層（Executor）を抽象化します。

### 1. 4層構造のアーキテクチャ定義
Kanon の構成を以下の4つの層に分離します。

```
+-----------------------------------------------------------------------------------+
|  【OS / PM 層】 Kanon Core (discord_agent.py / agent.py)                             |
|    - 役割: 長期記憶(Memory)の管理、タスクの起票・優先順位付け、人間との対話、成果の検収|
+-----------------------------------------------------------------------------------+
                                         │
                                         ▼ タスク発注 (TaskDescriptor)
+-----------------------------------------------------------------------------------+
|  【実行抽象層】 Executor Interface                                                 |
|    - 役割: OSからの発注を解釈し、適切な Worker を選択・起動して成果を OS へ返却する    |
+-----------------------------------------------------------------------------------+
                                         │
                                         ▼ 起動・アサイン
+-----------------------------------------------------------------------------------+
|  【Worker 層 (手足)】 AGY, Claude Code, Gemini CLI, etc.                            |
|    - 役割: 指示されたディレクトリ内で実際にファイルを編集・テストし、PRを作成する    |
|    - ※ 初期の実装（最初のWorker）として `agy` を利用する                           |
+-----------------------------------------------------------------------------------+
                                         │
                                         ▼ コミット / PR / 課題
+-----------------------------------------------------------------------------------+
|  【永続記憶・共同作業空間層】 GitHub                                                |
|    - 役割: プロジェクトの世界モデル、長期記憶、行動履歴（PR）、人間との対話（Review） |
+-----------------------------------------------------------------------------------+
```

### 2. 「発注・検収」によるタスク運用
* **発注 (TaskDescriptor)**: Kanon（OS/PM）は Worker に対し、単なるコマンドではなく「目的（Objective）」「制約（Constraints）」「期待成果・評価基準（Acceptance Criteria）」を定義した発注書を渡します。
* **検収 (Evaluation)**: Worker（手足）が作業を完了して成果（PRや差分）を返した後、Kanon（OS/PM）が `pytest` などの実行結果を自動検収し、「合格（マージ承認ボタンの提示）」か「リテイク（再実行）」かを判断します。

### 3. 「時間連続性」の担当
LLMやWorkerが「1セッション限りの専門家」であるのに対し、Kanonだけが「プロジェクトの時間軸（数週間〜数ヶ月の流れ）」を管理し、記憶を保持し続け、非同期に人間とコラボレーションする常駐プロセスとして責任を持ちます。

## 影響 (Consequences)
* **メリット**:
  * ファイルやGitの泥臭い操作を自前実装・デバッグする必要がなくなり、開発スピードが劇的に向上します。
  * 将来的に `Claude Code` など他の優秀なツールが登場した際にも、Kanon本体のコードを一切変更せず、ExecutorのWorker設定を差し替えるだけで容易に対応できます（極めて高いスケーラビリティ）。
  * 「Kanonはマネージャー」という役割が明確になり、教育やインボックス（仕事化）などの機能開発に注力できます。
* **デメリット**:
  * 実機環境において `agy` などの外部CLIが正常にインストールされ、ライセンスおよび認証情報が適切に設定されていることに依存します。
