## 1. 🗂️ 観点別設計レビュー

### 観点①: Memory の責務（保存するもの／保存しないもの）
* **現状の設計内容**:
  * 意思決定履歴、教訓（Lessons learned）、プロジェクトコンテキスト、ユーザーおよびエージェントのプロファイル・経歴情報、人物の特徴といった「長期的にエージェントの行動基準となる知識」を Markdown 形式で保存する。
* **未決定事項**:
  - 生の会話ログ（`conversation_history.json`）の保存期間とローテーション基準。
  - 失敗したタスクの一時ログや中間コマンド実行結果の扱い。
* **トレードオフ**:
  - **生ログ全保存**: 会話の詳細やニュアンスを完全に復元できるが、LLM コンテキストの肥大化と検索ノイズの増大を招く。
  - **要約のみ保存**: トークン効率は極めて高いが、特定の会話の具体的な一言やコンテキストの細部が失われる。
* **推奨案**:
  - `conversation_history.json` は「直近 N 往復（例: 30往復）」の生会話のみを維持するローテーション型とする。
  - それを超えた過去ログは Processor に送り、要約・教訓化した上で長期記憶 Markdown にマージし、生の会話履歴からは破棄する。一時的な中間コマンド実行結果やエラーログは長期記憶には保存せず、`agent.log` などの動作ログに残すのみとする。

---

### 観点②: Memory API
* **現状の設計内容**:
  * エージェントが記憶を操作・検索するためのインターフェース。
* **未決定事項**:
  - ツール（エージェントが LLM から呼び出す Function）のシグネチャと、背後の Python モジュール構造。
* **トレードオフ**:
  - **直接ファイル操作 (`read_file` / `write_file` の流用)**:
    新規ツールを増やす必要がなくシンプルだが、エージェントが Markdown や YAML Frontmatter のパース、Git コミットを手動で行うため、フォーマット崩れやパースエラーのリスクが極めて高い。
  - **専用 Memory API / ツール**:
    `store_memory()`, `query_memory()` などの構造化されたツールを提供。フォーマット制御や Git コミットがカプセル化されるため安全だが、ツール定義の複雑さが増す。
* **推奨案**:
  - 専用の Memory API ツール（例：`store_memory(category, title, content, tags)`, `search_memory(query, category)`）を定義する。エージェントは JSON 構造で引数を渡すだけでよく、Frontmatter の生成・更新、Git コミット、ファイル書き込みは API 内部で安全に実行されるように設計する。

---

### 観点③: Processor Pipeline (非同期・定時処理)
* **現状の設計内容**:
  * 「inbox ➔ processor ➔ output」の思考ログ処理パイプライン。
* **未決定事項**:
  - パイプライン（Processor バッチ）の起動トリガー（定時 Cron、一定トークン消費、セッション終了 handover 時など）。
  - エージェントのメイン稼働プロセスとのファイル競合対策。
* **トレードオフ**:
  - **定時バックグラウンド実行（完全非同期）**:
    ユーザーとのチャット中にエージェントの手を止めないが、エージェントが記憶を読み書きするタイミングと衝突し、Git やファイルロックの競合が発生する。
  - **セッション終了（Handover）時の同期実行**:
    競合は防げるが、セッション終了のタイミングで要約バッチが走るため、処理時間が増大する。
* **推奨案**:
  - 競合防止（生存最優先）のため、基本はエージェントの非アクティブ時間（例: 最終チャットから10分経過時）または `handover` 処理時に同期的に実行する「イベント＆アイドル起動モデル」を推奨。
  - ファイルロック機構を導入し、エージェント本体と Processor プロセス間の競合を排他制御する。

### 観点④: Directory Layout
* **現状の設計内容**:
  * `memory/` 配下に `inbox/`, `projects/`, `career/`, `health/`, `people/`, `decision_history/`。
* **未決定事項**:
  - 長期記憶 Markdown の命名規則（タイムスタンプベースか、連番 ID か）。
* **トレードオフ**:
  - **タイムスタンプ名 (`20260627_docker_proxy.md`)**:
    名前の衝突が起きにくく作成が容易だが、他のファイルからリンクする際（Docs as Code での参照時）にリンクが長くなり扱いにくい。
  - **シーケンシャルID名 (`DEC-009.md`)**:
    短く参照しやすいが、エージェントが複数スレッドや別環境で自律生成する際に ID 衝突（重複）が発生しやすい。
* **推奨案**:
  - **タイムスタンプ＋トピック名のハイブリッドファイル名**（例: `20260627_docker_proxy.md`）を採用し、ファイル名の衝突を避ける。
  - 連番IDや UUID などの一意な識別子は、YAML Frontmatter 内の `id` メタデータとしてプログラム的に管理する。

---

### 観点⑤: YAML Frontmatter Schema
* **現状の設計内容**:
  * YAML 形式のメタデータブロック。
* **未決定事項**:
  - スキーマの厳格性（カテゴリごとにメタデータフィールドを強制するか、自由記述にするか）。
* **トレードオフ**:
  - **スキーマの完全自由化**:
    エージェントが自由にフィールドを追加でき柔軟だが、プログラムパースや検索（タグ検索等）の信頼性が低下する。
  - **厳格なスキーマバリデーション (Pydantic 等)**:
    パースエラーを防げるが、エージェントが書き込み時にバリデーションエラーを起こすと自己修正できず、タスクがスタックするリスクがある。
* **推奨案**:
  - **共通スキーマ（id, title, category, tags, created_at, updated_at）のみを厳格にバリデーション**し、それ以外のカテゴリ固有メタデータ（例: career の `period` 等）は自由記述を許容する「半厳格スキーマ」を採用する。

---

### 観点⑥: Git を Memory Versioning として利用する境界
* **現状の設計内容**:
  * `memory/` ディレクトリ全体を Git で追跡し、自動コミットを行う。
* **未決定事項**:
  - メインコードリポジトリ（`kanon`）と同一リポジトリにするか、Git Submodule / 独立リポジトリに分離するか。
* **トレードオフ**:
  - **同一リポジトリでの管理**:
    構成がシンプルで初期構築が簡単だが、エージェントが記憶を更新するたびに大量の自動コミットがメインの Git 履歴を埋め尽くし、ノイズになる。また、エージェントにメインコード（`src/`）の変更権限を持たせたままで `push` させるため、セキュリティリスク（憲章の自己改変制限との競合）が高まる。
  - **Submodule または独立リポジトリ化**:
    エージェントに与える push 権限を `memory/` リポジトリのみに限定（最小権限の原則）でき、コードのコミットログも汚れないため安全。ただし、リポジトリが2つに分かれるため初期セットアップや環境構築の手間が増える。
* **推奨案**:
  - **【極めて重要】`memory/` は独立した Git リポジトリ（または Git Submodule）として物理的に分離する。**
  - エージェントコンテナ内の Git には `memory` リポジトリに対する push 権限のみを付与し、メインコードの `src/` があるリポジトリへの直接プッシュ権限からは隔離する。これにより生存性とセキュリティを担保する。

---

### 観点⑦: Retrieval / Search の基本方針
* **現状の設計内容**:
  * VectorDB は導入せず、インメモリ ＋ 永続ファイルでのシンプルな会話履歴。
* **未決定事項**:
  - 長期記憶 Markdown の検索方式（LLM に全ロードするか、grep 等のローカル検索を挟むか、Vector 検索か）。
* **トレードオフ**:
  - **全ロード方式**: LLM が全コンテキストを理解して応答できるが、ファイルが増えるとコンテキスト上限を超え、API コストが破綻する。
  - **キーワード検索/Grep 方式**: 軽量・高速・低コストでローカル完結するが、文脈や同義語による意味的（Semantic）検索ができない。
  - **Vector 検索 (VectorDB/Embeddings)**: 意味的検索が可能だが、外部 API (Gemini Embedding) への依存、VectorDB インフラの導入が必要になり、ローカル生存性（生存最優先の原則）を脅かす。
* **推奨案**:
  - Phase 2 初期フェーズでは **Grep ツール ＋ Frontmatter タグ検索のハイブリッド（軽量ローカル検索）**を推奨。
  - ファイル数が数百〜数千件に増大した段階で、ローカル完結する **SQLite-FTS5（全文検索）**、あるいは外部依存を排した **ローカル Vector ファイル (Faiss + 軽量オンプレミス Embedding モデル)** へ段階的にアップグレードする。

---

### 観点⑧: 将来の Digital Twin への拡張ポイント
* **将来のビジョン**:
  * ユーザーの過去の行動パターン、経歴、意思決定の歴史から、ミラーリング（ユーザーならどう判断するか）をシミュレーションする。
* **拡張ポイントの設計**:
  - `memory/career/` および `memory/decision_history/` の Markdown ファイル群から、ユーザーの「価値観・判断軸プロファイル」を抽出し、プロンプトに動的インジェクションする機構。
  - ユーザーから受け取ったフィードバックや修正（例: `/learn` コマンドで蓄積した内容）を `memory/lessons/` に集約し、プロファイルへ自動フィードバックループを回す構造。

## 2. 📝 ADR (設計決定レコード) 起票候補

Phase 2 開始にあたり、合意形成が必要な重大なアーキテクチャ上の決定事項です。

1. **ADR-010: `memory/` ディレクトリの独立 Git リポジトリ化による権限最小化**
   - **内容**: 記憶の頻繁な更新による Git ログの汚染を防ぎ、エージェントの push 権限を memory のみに絞るため、記憶ディレクトリを Git Submodule または別リポジトリとして分離する決定。
2. **ADR-011: ローカルファーストな記憶検索エンジンの段階的採用方針**
   - **内容**: 外部 VectorDB への依存によるインフラ複雑化を回避し、生存最優先の原則に基づいて「Grep/Tag ➔ SQLite-FTS5 ➔ ローカル Vector ファイル」へと段階的に検索機構をスケールアップしていく決定。
