# Phase 1 実装計画書 (Phase 1 Implementation Plan)

本ドキュメントは、技術ロードマップ Phase 1 (Survival & Observation / 生存・観測・復旧基盤) の構築に向けた、具体的な実装タスクの分解と設計書です。

---

## 🛠️ タスク分解と設計

### ① Validation Engine (検証エンジン)

エージェントが自己のソースコードを変更する前に、別プロセスでその整合性を検証するエンジン。

#### Task-001: 構文チェック機能 (Syntax Verifier)
* **目的**: 適用対象の Python ソースコードが文法的に正しいかを検証する。
* **責務**: `py_compile.compile()` または AST (抽象構文木) 解析を用い、指定ファイルの構文エラーの有無およびそのエラー内容を検出する。
* **依存関係**: なし（標準ライブラリ）
* **配置場所**: `ai-agent/workspace/src/utils/validator.py`
* **インプット**: `file_path` (str: 検査対象ファイルのパス)
* **アウトプット**: `(success: bool, error_message: str | None)`
* **テスト方法**: 意図的に括弧の閉じ忘れや不完全な定義を記述した候補ファイルを別名で保存し、本関数を実行して正しく `success=False` とエラー文が返るかテストする。
* **Copilot委譲**: **可能 (100%)**

#### Task-002: インポートテスト機能 (Import Verifier)
* **目的**: コード実行時に依存ライブラリの不足等による実行時エラー（ImportError等）を事前に防ぐ。
* **責務**: 対象ファイルを Python サブプロセスとして仮想ロード（インポート）し、依存関係の解決とトップレベル定義のロードが正常か検証する。
* **依存関係**: Task-001 (構文エラーがないこと)
* **配置場所**: `ai-agent/workspace/src/utils/validator.py`
* **インプット**: `file_path` (str)
* **アウトプット**: `(success: bool, error_message: str | None)`
* **テスト方法**: `import non_existent_package` などの存在しないインポートコードが書かれたファイルを入力し、正しくエラーとしてトラップできるか検証。
* **Copilot委譲**: **可能 (100%)**

### ② Healthcheck (健康診断)

コンテナの生存、およびホストPCのリソース状態を公開し監視可能にする仕組み。

#### Task-003: リソース状況取得 (Resource Monitor)
* **目的**: CPU・メモリ・ディスクリソース状況をJSONで取得する。
* **責務**: 標準ライブラリ (`os`, `shutil`) や必要に応じて `psutil` を用いて、ホスト・コンテナ上の負荷状況を検出する。
* **依存関係**: なし
* **配置場所**: `ai-agent/workspace/src/utils/healthcheck.py`
* **インプット**: なし
* **アウトプット**: `{ "cpu_percent": float, "memory_free_mb": int, "disk_free_gb": float, "timestamp": "ISO-8601" }`
* **テスト方法**: 単体で実行し、期待されるキーと値を持つJSON/辞書が返却されることを確認する。
* **Copilot委譲**: **可能 (90%)**

#### Task-004: Flask ヘルスチェックエンドポイント (Healthz Endpoint)
* **目的**: 監視Controllerがコンテナの生存・ヘルス状態をポーリングできるようにする。
* **責務**: Flaskサーバに `/healthz` ルートを追加し、Task-003の結果およびエージェントの現在ステータスを返却する。
* **依存関係**: Task-003
* **配置場所**: `ai-agent/workspace/src/discord_agent.py`
* **インプット**: HTTP GET `/healthz`
* **アウトプット**: HTTP レスポンス (HTTP 200/500) および状態JSON
* **テスト方法**: コンテナ内のFlaskサーバ起動後、`curl http://localhost:5000/healthz` を実行してHTTPステータスとJSON構造を検証する。
* **Copilot委譲**: **可能 (100%)**

### ③ Rollback Monitor (ロールバックモニター)

ホスト/Controller側でコンテナの死活を常時監視し、異常検知時に自動で復旧（ロールバック）するスクリプト。

#### Task-005: 監視ループスクリプト (Heartbeat Monitor)
* **目的**: エージェントコンテナの `/healthz` に対する定期的な死活監視。
* **責務**: 定期的（例: 15秒間隔）に `/healthz` へポーリングし、連続失敗数またはタイムアウトを検知して異常フラグを立てる。
* **依存関係**: Task-004
* **配置場所**: `controller/monitor.py` (またはホスト側で実行する監視スクリプト)
* **インプット**: なし (タイマーループ)
* **アウトプット**: 監視ログ出力、および異常検知時のロールバックトリガー呼出
* **テスト方法**: エージェントコンテナを意図的に停止、または500エラーを返す状態にし、スクリプトが即座に異常を検知することを確認する。
* **Copilot委譲**: **可能 (80%)**

#### Task-006: ロールバック復元機能 (Rollback Executer)
* **目的**: 起動失敗時、`backups/` にある LKG バージョンを復元してコンテナを再起動させる。
* **責務**:
  * 異常検知時、`ai-agent/workspace/backups/` から直近の正常動作ファイルを `ai-agent/workspace/src/` 配下に上書きコピー。
  * `docker restart ai_agent_core` などのホスト側Dockerコマンドを呼び出す。
* **依存関係**: Task-005
* **配置場所**: `controller/monitor.py` (または `controller/rollback.py`)
* **インプット**: なし (モニターからの呼び出し)
* **アウトプット**: ロールバック適用ログ、コンテナの再起動実行
* **テスト方法**: `src/` 配下に意図的にエラーのあるコードを置いて再起動し、監視スクリプトが自動で `backups/` から正常ファイルを書き戻してコンテナを起動完了させるか検証する。
* **Copilot委譲**: **一部可能 (Dockerコマンドのシェル連携やパス指定はAGY/人間が確認)**

### ④ Unit Test作成

実装したコンポーネントが常に正常動作することを確認するための自動テストスイート。

#### Task-007: 検証エンジンおよびヘルスチェックのテストコード作成
* **目的**: validator および healthcheck のテストカバレッジを確保する。
* **責務**: `pytest` に対応した正常系・異常系のテストケースを記述する。
* **依存関係**: Task-001, Task-002, Task-003, Task-004
* **配置場所**: `ai-agent/workspace/tests/test_validator.py`, `ai-agent/workspace/tests/test_healthcheck.py`
* **インプット**: なし (テストランナーの実行: `pytest`)
* **アウトプット**: テスト結果レポート (Pass/Fail)
* **テスト方法**: コンテナまたはローカル環境で `pytest tests/` を実行し、全テストケースが合格することを確認。
* **Copilot委譲**: **可能 (100%)**

### ⑤ Logging整備

エージェントの活動やControllerの監視状態をファイルに構造化して永続化する。

#### Task-008: ログファイルローテーションとダッシュボード統合
* **目的**: 動作ログをファイル保存し、世代管理（ローテーション）を行う。
* **責務**: Pythonの `logging` モジュールを用い、`workspace/state/agent.log` にタイムスタンプ付きの構造化されたログを出力する。
* **依存関係**: なし
* **配置場所**: `ai-agent/workspace/src/utils/logger.py`
* **インプット**: ログ出力リクエスト (メッセージ、ログレベル)
* **アウトプット**: ローテーションされたログファイル (`agent.log`, `agent.log.1` 等)
* **テスト方法**: 大量のログを意図的に出力させ、ファイルサイズ上限（例: 1MB）に達した際に正常にファイル分割（ローテーション）が行われるかを検証。
* **Copilot委譲**: **可能 (100%)**

### ⑥ Docker周辺

ディレクトリ構造再編成後のビルド整合性の確保。

#### Task-009: Dockerコンテナ・ローカル起動・パーミッション確認
* **目的**: 境界再設計後のマウントパスおよび権限の検証。
* **責務**:
  * ホスト側から `docker-compose build` および `docker-compose up` を実行する。
  * `workspace/state/` や `workspace/backups/` 等のディレクトリに対するコンテナからの読み書き権限を確認。
* **依存関係**: 全タスク
* **配置場所**: ホスト側の環境
* **インプット**: なし
* **アウトプット**: コンテナの稼働ステータス
* **テスト方法**: コンテナを正常に起動し、ホストとコンテナ間でマウントされたファイルの読み書きがエラーなく実行できるか（Permission Deniedの有無）を確認。
* **Copilot委譲**: **不可 (実機確認が必要なため、AGYまたは人間が担当)**
