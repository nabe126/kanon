# 意思決定履歴 (Architecture Decisions)

本プロジェクトにおける主要な設計・アーキテクチャ上の決定事項を記録します (Architecture Decision Records)。

---

## [ADR-001] Docs as Code 運用の導入

### ステータス
**承認済** (2026-06-19)

### 背景 (Context)
外部のNotionやメモ帳などに仕様や計画をまとめようとすると、「整理する人がいない」状態になり、最終的にドキュメントとコードが乖離して破綻する（運用コスト過多）。
人間は「メモを書く」ことはできるが、「後で分類・整理する」作業は敬遠しがちである。

### 決定 (Decision)
ドキュメントをすべてリポジトリ内の `docs/` ディレクトリにMarkdownとして保存し、コードと同等にGit管理する「Docs as Code」を採用する。
情報の整理、ドキュメントの更新案の作成はすべてAI（アシスタントおよび自律エージェント）が主導して行い、人間は「レビューおよび承認」のみを担当する運用とする。

### 影響 (Consequences)
* メリット：ドキュメントがコードの変更と同期しやすくなり、常に最新の仕様が維持される。
* メリット：AIが事前にドキュメントを読み込むだけでプロジェクトの背景を100%把握できる。
* デメリット：ドキュメント更新に伴うGitコミットが増加する。

## [ADR-002] Dockerコンテナの自己再起動によるコード反映

### ステータス
**承認済** (2026-06-19)

### 背景 (Context)
エージェントが自己のソースコード（Python）を変更した際、Pythonプロセスをメモリ上で動的にリロードするのはメモリリークや未定義動作の原因になりやすく、確実性が低い。

### 決定 (Decision)
エージェントはDockerコンテナ（`ai_agent_core`）内で動作させ、ホストの `/var/run/docker.sock` をコンテナにマウントする。
自己改変を行った後は、コンテナ内からDocker API（またはCLI）経由で `docker restart ai_agent_core` を呼び出し、コンテナごと再起動することで最新のコードを確実にロードする。

### 影響 (Consequences)
* メリット：クリーンな状態から新しいコードを実行できるため、実行の安定性が高い。
* デメリット：再起動時に構文エラーやDiscord接続エラーがあった場合、エージェントが二度と立ち上がらなくなり、制御不能（デッドロック）になる。安全対策（ロールバック、事前検証）が必須。

## [ADR-003] LLM Provider の抽象化設計

### ステータス
**承認済** (2026-06-19)

### 背景 (Context)
初期実装はGoogle Gemini (gemini-2.5-flash) 専用の記述になっているが、将来的にOpenAI (GPT-4oなど) や Anthropic (Claude 3.5 Sonnetなど) に柔軟に切り替え、あるいは併用できるようにしたい。

### 決定 (Decision)
エージェントコアが直接特定のLLM SDKを呼ぶのではなく、仲介する抽象的な「Providerレイヤー」を設ける。
エージェントはProvider名やモデル名などの詳細パラメータを知らず、共通化された `generate_content` や `call_tool` のインターフェースを通じてLLM機能を利用する。

### 影響 (Consequences)
* メリット：他のLLMへの拡張が非常に容易になり、Gemini障害時の自動フォールバック先としても他社LLMを使いやすくなる。
* デメリット：LLM独自の固有機能（特定のパラメータ指定など）を使いにくくなるため、共通化処理の設計が必要。

## [ADR-004] Gemini API 503エラー対策の導入

### ステータス
**承認済** (2026-06-19)

### 背景 (Context)
Gemini APIなどのクラウドLLMは、稀に 503 (Service Unavailable) エラーやレート制限によるエラーを返却することがある。自律稼働しているエージェントがこの一過性のエラーで即座にクラッシュ・停止するのは望ましくない。

### 決定 (Decision)
LLM Providerレイヤーでのリクエスト送信処理において、指数バックオフによるリトライ処理（1秒, 2秒, 4秒, 8秒, 16秒の間隔で最大5回リトライ）を組み込み、一時的なAPIエラーに対して回復性（Resilience）を持たせる。

## [ADR-005] 生存最優先 (Survival First) 原則の制定

### ステータス
**承認済** (2026-06-19)

### 背景 (Context)
機能追加やインテリジェンスの向上（高性能化）を追求するあまり、複雑性が増し、回復不能なバグやクラッシュでエージェントが「死ぬ」リスクが高まってしまう。

### 決定 (Decision)
開発・運用の全フェーズにおいて、「生存 > 観測 > 復旧 > 改善 > 高性能化」の優先順位を遵守し、**「壊れた時に確実に元に戻せるか？」**を新機能追加の唯一最大の判断基準とする。戻せない場合は実装を見送る。
