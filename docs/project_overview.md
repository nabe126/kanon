# Kanon (arahabaki) Project Overview & Context

このドキュメントは、外部のAI（ChatGPTやGeminiなど）に本プロジェクトの全体像、現在の稼働状況、アーキテクチャ、および今後の開発ロードマップを瞬時に同期するための「総合引き継ぎ用マスターコンテキスト」です。このファイルの内容をコピペまたはファイル添付して読ませるだけで、外部AIはプロジェクトの全容を完全に把握できます。

## 🔍 1. プロジェクト概要 (What is Kanon?)

* **開発コード**: `arahabaki`
* **プロダクト名**: `Kanon`
* **コアコンセプト**: 
  Kanonは、単に指示されたコードを書くだけのAIではありません。**「人間と協調しながら、自ら課題を発見し、プロジェクトを主体的に育てる自律型開発パートナー（AI）」** を目指しています。

### 🧠 基本哲学（GitHub ＝ AIの外部脳）
Kanonは、GitHubの各種機能を「ソースコードの置き場所」ではなく「自身の長期記憶・認知インターフェース」として活用します。
* **Issue** ➔ **未来の仕事（タスク・課題）**
* **Pull Request (PR)** ➔ **実施内容（行動履歴）**
* **Review** ➔ **人間からの教育・フィードバック**
* **Projects** ➔ **現在の脳（タスクの優先順位と状態）**

## 🏛️ 2. 現在のアーキテクチャと動作環境

実機（GPDホスト）上でDockerコンテナを起動し、Kanonを「PM (プロジェクトマネージャー/OS)」、外部ツールを「Worker (手足)」として分離した実行アーキテクチャを採用しています。

```
+-----------------------------------------------------------------------------------+
|  【OS / PM 層】 Kanon Core (discord_agent.py / agent.py)                             |
|    - 役割: 長期記憶(Memory)の管理、タスクの起票・優先順位付け、人間との対話、成果の検収|
+-----------------------------------------------------------------------------------+
                                         │
                                         ▼ タスク発注 (TaskDescriptor)
+-----------------------------------------------------------------------------------+
|  【実行抽象層】 Executor                                                           |
|    - 役割: OSからの発注を解釈し、最適な Worker を選択・起動して成果を OS へ返却する    |
+-----------------------------------------------------------------------------------+
                                         │
                                         ▼ 起動・アサイン
+-----------------------------------------------------------------------------------+
|  【Worker 層 (手足)】 AGY (Antigravity), Claude Code, etc.                          |
|    - 役割: 指示されたディレクトリ内で実際にファイルを編集・テストし、PRを作成する    |
|    - ※ 初期の実装（最初のWorker）として `agy` を利用する                           |
+-----------------------------------------------------------------------------------+
                                         │
                                         ▼ コミット / PR / 課題
+-----------------------------------------------------------------------------------+
|  【永続記憶・共同作業空間層】 GitHub                                                |
|    - 役割: プロジェクトの世界モデル、長期記憶、行動履歴（PR）、人間との対話（Review） |
+-----------------------------------------------------------------------------------+
```

### 🔒 安全ゲート (ASEP) とセキュリティ
* **ASEP (承認ゲート)**: コマンド実行（Workerのキック）前には必ず `[✅ 承認]` `[🛑 却下]` ボタンを提示し、人間がクリックしない限り実行されません。
* **セキュリティブロック**: 機密情報（`secrets/`, `.env`）やシステム監視スクリプト（`controller/`）、および Git 管理領域（`.git/`）へのアクセスを物理的にブロックするガードレールが実装されています。

## 🚦 3. 現在の稼働ステータス (Current State)

* **実機検証（L3フェーズ）完全合格**:
  * Discord上で「`make status` を実行して」とメンションすると、エージェントが自動的に実行計画（PLAN）を起票。
  * Discordチャットに **`[✅ 承認 (YES)]` `[🛑 却下 (NO)]`** のボタンを自動添付して返信。
  * 人間が `[承認]` をクリックすると、非同期でコンテナ内からホスト上のコマンドが走り、コンテナやポートのバインド稼働状況（STDOUT）が自動でチャットに返却される一連のループが100%正常動作しています。
* **ローカルCLI REPL (`agent.py`) の自律エージェント化**:
  * コンテナ内で `python3 src/agent.py` を実行するだけで、コンテナに入りながらCLI上で同一の自律開発ツールを使用できます。

## 🗺️ 4. 開発ロードマップ (6 Sprints)

Kanonが自律的に学習し仕事を作る「自己拡張ループ」に向けて、以下の6段階のスプリントで開発を推進します。

1. **Sprint 1 (実行基盤)**: **Kanon-AGY連携 (脳と手の接続)**
   * 自前でのGit/GitHub操作実装を廃止し、既存の強力な `agy` CLIを実行エンジンとして利用。Kanon（PM）から `agy "指示"` をアサインし、自律的にブランチ作成・コード編集・テスト・プッシュ・PR起票までを実行させる接続基盤の構築。
2. **Sprint 2 (仕事化)**: **Task Manager**
   * AIが「今やる仕事（タスク）」を管理・優先順位付けするタスク管理機能（タスクキュー）。
3. **Sprint 3 (監視・観測)**: **Observer**
   * Discord、ログ、Memory、pytest結果などをバックグラウンドで常時読み取る仕組み。
4. **Sprint 4 (プランニング)**: **Planner**
   * `Inbox`（対話ログ、メモ、PDFなどの流入ゲートウェイ）を自動解析し、自ら「Issue（未来の仕事）」を自動起票する仕組み。
5. **Sprint 5 (実作業)**: **Executor**
   * タスクの性質や得意分野に応じて、AGYや他のモデル（Claude等）にタスクを自動で割り振る機能。
6. **Sprint 6 (学習・自己改変)**: **Reflection**
   * タスク完了時の振り返りを `Memory` へ蓄積し、教育の第一層（憲法・設計）を自ら洗練する学習機能。

## 🚀 5. 次回行うタスク (Sprint 1: Kanon-AGY連携)

### 1. タスク発注記述（TaskDescriptor）の定義
* KanonからExecutorへ渡す発注書のフォーマット（目的、制約、評価基準）をJSON/Markdown形式で定義。

### 2. Executor による agy (Worker) の起動インターフェース実装
* 発注書を解釈してコンテナ内の `agy` CLIに「指示・制約」を渡して自律実行させる連携ロジックの実装。

### 3. 検収ロジック（テスト実行と評価判定）の実装
* Worker（`agy`）が完了した成果物を受け取り、自動で `pytest` 等を走らせて合格/不合格を判定し、人間に最終承認（マージボタン）を提示する検収プロセスの実装。
